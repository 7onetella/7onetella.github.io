How to better manage your ECS services in AWS cloud

Motivation

If you landed at my page from google search, then I am going to assume that you looking for better way to manage your ECS services. If anyone has to look at CloudFormation templates all day and compare revisions, compare with the ones you copied and pasted from, I can tell you that it’s a daunting task from my own experience. It used to make my head hurt. Writing cloudformation template is certainly important but it’s not fun. The only time I had fun with CloudFormation template was when I was not writing it but achiving the same result through automation.

Approach #1

CloudFormation templates all the way

If that’s what you are interested in, then take a look at my other post here.

Approach #2

Combination of CloudFormation templates and automation scripts. This is what I call the fun way. Continue to read this blog post if you want to hear about how I would have fun deploying ECS services all day and every day.

The phrase, infrastructure as code has been thrown around a lot. I wonder how many Ops / Devs get to really do hard core infrastructure as code.

The wrong way to do it

  • You have large individual CF templates for services which includes cluster, alb, and services all in one.
  • CloudFormation templates have huge number of input parameters.
  • Giving resources explicit names in places you shouldn’t.
  • Your input parameters are asking the users to enter data users don’t have a clue about.
  • Unrelated AWS resources are grouped together.
